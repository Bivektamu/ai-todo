## Risk posture: Cleared for merge

The migration works end to end — CRUD passes against live Neon, all tests are green. Both must-fix items are resolved: error handling is in place (try/catch in actions.ts, error.tsx boundary), and stale generated files are removed. The should-harden items (connection failure ergonomics, cold start readiness) are real but tolerable for a single user todo app at this scale.

## Must fix before merge (2)

### M1 ✅ RESOLVED: No error handling on Prisma calls — `app/actions.ts`, `app/page.tsx`

Every Prisma call in `actions.ts` (create, findUnique, update, delete) and `page.tsx` (findMany) is a bare await with no try/catch. If Neon drops the connection mid request (pooler restart, network blip, or the database being paused), the user sees an unhandled rejection that Next.js surfaces as a raw error page. A server component crash on `page.tsx` also takes down the entire page, not just the failing query.

**Fix**: add an `error.tsx` boundary at `app/error.tsx` so the page degrades to a friendly "something went wrong" state instead of showing a stack trace. For actions, wrap Prisma calls in try/catch and return a structured result (or at minimum, log the error and let the revalidatePath still fire so the page reloads with stale data rather than crashing).

**File:line**: `app/actions.ts:13,27,35,50` and `app/page.tsx:11`

### M2 ✅ RESOLVED: Stale generated Prisma files tracked in git — `app/generated/app/generated/`

The old schema had `output = "../app/generated/prisma"` which placed generated files inside `app/generated/app/generated/`. The new schema uses `output = "."` so the client lives at `app/generated/prisma/`. The inner directory (`app/generated/app/generated/prisma/`) is stale generated output but is still tracked in git, including a stale `schema.prisma` with the old `sqlite` provider.

**Fix**: `git rm -r app/generated/app/generated/` and add `app/generated/` to `.gitignore` (generated files should not be tracked; they're rebuilt by `prisma generate`). If the generated client must be tracked in this project, delete only the inner stale directory.

**File:line**: `app/generated/app/generated/prisma/schema.prisma` and sibling files

## Should harden (4)

### S1: Non-null assertion on DATABASE_URL produces cryptic crash — `lib/prisma.ts:7`

`process.env.DATABASE_URL!` uses a non-null assertion. If the variable is missing, the `PrismaNeon` constructor receives `undefined` and throws with an internal error ("No database host or connection string was set..."). The ADR correctly says the app should fail fast, but the error message doesn't tell the operator which variable is missing.

**Fix**: add an explicit guard before constructing the adapter:

```ts
if (!process.env.DATABASE_URL) {
  throw new Error("DATABASE_URL environment variable is required");
}
```

This gives a clear, actionable message and costs one line.

### S2: Module level initialization blocks entire app on cold start — `lib/prisma.ts:6-8`

`new PrismaNeon({...})` runs at import time. In a serverless deployment every cold start establishes a new connection. If Neon is unreachable (regional outage, database paused after idle), the entire `lib/prisma.ts` module fails to load and every route that imports it returns 500. There's no retry, no lazy initialization, no circuit breaker.

**Fix**: consider lazy initialization so the database connection is established on first use, not at import time. This lets the health check endpoint (if any) or static routes remain available even when the database is temporarily down. For a single page todo app the blast radius is total, and a lazy init+retry pattern keeps the page serving (even if empty) during brief Neon blips.

### S3: No pool configuration for serverless concurrency — `lib/prisma.ts:6`

The Neon connection string uses the pooled endpoint (`-pooler`), which is correct. But the Prisma adapter doesn't expose pool tuning — there's no `connectionLimit`, no `idleTimeoutMillis`, no `maxUses`. Under concurrent serverless invocations (multiple users hitting the page simultaneously), the connection pool on Neon's side is the only throttle. If it saturates, requests queue or fail. For a single user app this is low risk; for a deployed app it becomes real fast.

**Fix**: document the expected concurrency limits in the ADR or a README. If the app scales beyond a handful of users, consider `@neondatabase/serverless` directly (captured as an alternative in the ADR) for fine grained pool control, or configure the Neon project's pooler settings in the Neon dashboard.

### S4: No `error.tsx` boundary for server component failures — `app/` root

Next.js App Router renders `page.tsx` as a server component. If `prisma.todo.findMany()` throws (Neon unreachable, query timeout), there's no `error.tsx` to catch it. The user sees Next.js's default error overlay in dev or a generic 500 in production. Adding an error boundary gives a degraded but usable page.

**Fix**: create `app/error.tsx` with a "client" directive that renders a friendly message and a retry button. This is the companion to M1 — M1 handles the action layer, S4 handles the server component layer.

## Watch / accept (3)

### W1: `as never` cast on PrismaClient constructor — `lib/prisma.ts:10`

The ADR documents this as a known Prisma 7 adapter typing workaround. It's fragile: if Prisma 7 changes its constructor signature in a point release, this cast silently masks the mismatch until runtime. Accept for now but add a version pin comment or revisit when Prisma 7 stabilizes its adapter types.

### W2: No read replica or fallback — design choice

The app has a single Neon connection string. If that Neon project is paused, deleted, or goes down, the app is dead. There's no read replica, no fallback database, no local SQLite fallback. For a single page todo app this is an acceptable tradeoff; document it in the ADR's Consequences section.

### W3: `sslmode=require&channel_binding=require` — connection string strictness

The connection string requires SSL with channel binding. If the Node.js TLS stack or the Prisma adapter doesn't support channel binding (rare but possible in constrained runtimes like edge functions), the connection fails. The ADR correctly chose the Neon adapter (not edge compatible per the `index-browser.js` warnings). Accept as correct security posture for a Node.js runtime.
