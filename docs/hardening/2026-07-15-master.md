# Hardening: Drag and drop reorder

Date: 2026-07-15
Branch: master
ADR: 005-drag-and-drop-reorder
Tests: configured (vitest + @testing-library/react + playwright, 61 unit, all green)

## Risk posture: Harden before merge

The reorder feature works end to end: the drag UI is smooth, the sortOrder field is backfilled, and all 61 tests pass. But the server action has two TOCTOU (time of check to time of use) races that can produce duplicate sortOrder values or silently lost reorders under concurrent use. Neither requires multiple users: two browser tabs, a fast double drag, or a create during a reorder are enough to trigger them. Both are fixable with small, contained changes.

---

## Must fix before merge (2)

### M1: TOCTOU race on sortOrder in createTodo — `app/actions.ts:35-44`

`createTodo` computes `sortOrder = max(sortOrder) + 1` via an `aggregate` call, then uses that value in a separate `create` call. Two concurrent creates (two tabs submitting at the same moment) can both read the same `maxOrder`, both compute the same `sortOrder`, and insert two todos with identical sortOrder values. This breaks the invariant that sortOrder is unique and sequential.

The same race exists between a `createTodo` and a `reorderTodo`: if a create reads the max while a reorder is renumbering everything, the new todo gets a sortOrder equal to one of the renumbered values.

**Fix**: wrap the aggregate + create in a Prisma interactive transaction so the read and write are atomic:

```ts
const result = await prisma.$transaction(async (tx) => {
  const maxOrder = await tx.todo.aggregate({ _max: { sortOrder: true } });
  const sortOrder = (maxOrder._max.sortOrder ?? -1) + 1;
  return tx.todo.create({
    data: { title: title.trim(), priority, dueDate, sortOrder },
  });
});
```

This guarantees the aggregate and create see a consistent snapshot. The `$transaction` with a callback uses Prisma's interactive transactions (proper isolation), unlike the array form used in `reorderTodo`.

**File:line**: `app/actions.ts:36-44`

### M2: TOCTOU race in reorderTodo: read outside transaction — `app/actions.ts:131-159`

`reorderTodo` reads all todos via `findMany` at line 131, then enters a transaction at line 152 to apply the new sortOrder values. Any mutation that lands between those two lines (another reorder, a create, or a delete) creates a stale write:

- **Concurrent reorder**: both read the same starting order, both compute a new order, the second transaction overwrites the first. One user's drag is silently lost.
- **Concurrent delete**: the deleted todo is still in the `todos` array from the stale `findMany`. The transaction's `update` for that id fails with a "record not found" error, rolling back the entire transaction. The reorder silently fails.

Since `$transaction` with an array of operations does not wrap the reads, the gap is unprotected. Interactive transactions (`$transaction(async (tx) => {...})`) would close this gap but require all reads to use the transaction client.

**Fix**: wrap the read + reorder logic in an interactive transaction:

```ts
await prisma.$transaction(async (tx) => {
  const todos = await tx.todo.findMany({ orderBy: { sortOrder: "asc" } });
  const draggedIndex = todos.findIndex((t) => t.id === parseInt(id, 10));
  if (draggedIndex === -1) return;
  // ... splice logic ...
  for (let i = 0; i < todos.length; i++) {
    await tx.todo.update({
      where: { id: todos[i].id },
      data: { sortOrder: i },
    });
  }
});
```

Note: this changes from the array form of `$transaction` (which batches operations but doesn't provide a transactional client) to the callback form (which provides `tx` with snapshot isolation). The array form is documented as "sequential operations with no transactional client"; the callback form is the only way to get proper read-your-writes isolation.

Alternatively, if the array form must be kept, add a `version` or `updatedAt` optimistic lock to each todo and use a conditional update, but this is more invasive.

**File:line**: `app/actions.ts:131-159`

## Should harden (2)

### S1: Silent failure on reorder: user gets no feedback — `app/actions.ts:160-164`

When `reorderTodo` fails (caught at line 160), the error is only logged to the console. The `revalidatePath` still fires at line 164, so the page refreshes to the old state. The drag appears to "snap back" with no explanation. The user has no way to know whether their reorder succeeded or failed.

The same pattern exists in `createTodo` (lines 44-48): a failed create still revalidates, so the page refreshes but the new todo never appears. This is less severe because the form input is still visible, but it is the same class of issue.

**Fix**: for `reorderTodo`, consider using `useOptimistic` in the client so the UI updates immediately while the server processes. On failure, the optimistic update rolls back naturally. For a simpler fix: return a result object from the action (e.g. `{ success: boolean }`) and let the client handle the error case — but this requires changing from the `action` prop pattern to `useActionState` or a manual `fetch`.

At minimum, move `revalidatePath` inside the try block so a failed reorder doesn't trigger an unnecessary page refresh:

```ts
try {
  // ... reorder logic ...
  revalidatePath("/");
} catch (error) {
  console.error("Failed to reorder todo:", error);
}
```

**File:line**: `app/actions.ts:160-164`

### S2: Transaction size scales linearly with list — `app/actions.ts:152-159`

Every reorder issues N individual `UPDATE` statements inside a single transaction. For 50 todos this is fine. For 500, the transaction holds row locks for the duration of 500 sequential queries. For 5000, it likely exceeds PostgreSQL's `statement_timeout` or Prisma's internal batch limits. The ADR acknowledges this in Follow-up: "per-reorder full renumbering could be replaced with gap-based ordering."

No fix needed now for a single user todo app with tens of items, but document the ceiling so the next engineer knows when to switch strategies. The gap-based approach (assign sortOrder 1000, 2000, 3000, ... and rebalance only when gaps exhaust) is the standard solution.

**File:line**: `app/actions.ts:152-159`

## Watch / accept (3)

### W1: handleReorder closure captures stale fullCustomSorted — `components/TodoFilter.tsx:82-113`

`handleReorder` is wrapped in `useCallback` with `[fullCustomSorted]` as the dependency. When the user drags, `fullCustomSorted` reflects the list at the time of the last render. If a mutation (create, delete, another reorder) lands between the render and the drag-end event, the `targetId` is computed from stale data. The server side reorder still uses fresh data (`findMany` in the action), so the outcome is mostly correct, but the target position might not match what the user saw when they dropped.

**Accept**: React's batching and the synchronous nature of form submissions make this window extremely small. The server side reorder is the source of truth. If it becomes a problem, switch to `useOptimistic` with a pending reorder state.

### W2: parseInt allows partial numeric matches — `app/actions.ts:135,145`

`parseInt("123abc", 10)` returns `123`, not `NaN`. A crafted `targetId` of `"123<script>"` would match todo id 123 and execute a reorder targeting that todo. This is not a security issue (no injection possible) but is slightly imprecise input validation. `Number(id)` would return `NaN` for `"123abc"`, which is cleaner. However, `id` and `targetId` come from `formData.get()`, which in normal use receives the `toString()` of a number. An attacker would need to bypass the form.

**Accept**: the existing validation (`typeof id !== "string"` and `parseInt` returning `NaN` for non-numeric strings) covers the practical cases. Switch to `Number()` or a regex if the precision becomes important.

### W3: No deduplication or idempotency on reorder — design choice

If the same reorder is submitted twice (double click on drag handle, or a retry after a network blip), it processes twice. The second execution reorders an already-reordered list and produces a different result than intended. There is no idempotency key or request deduplication.

**Accept**: the drag-and-drop interaction makes double submission unlikely (the user would need to release and re-grab in the same position). Server actions also benefit from React's built-in form submission deduplication. If idempotency becomes important, add a `requestId` or `lastReorderId` to the todo model.

## Verification

Run a concurrency smoke test for the reorder action by submitting two reorder requests simultaneously:

1. Seed 5 todos in the database.
2. Open two tabs, both in Custom sort mode.
3. In tab 1, drag todo A after todo C. In tab 2, drag todo B after todo D.
4. Refresh both tabs. Verify that both reorders took effect and no sortOrder values are duplicated.

For the create race (M1), open two tabs, type a title in each, and submit both forms at the same time. Verify both todos appear with sequential, non-duplicate sortOrder values in Custom sort mode.

For the delete during reorder race (M2), start a reorder and quickly delete a different todo before the reorder completes. Verify the list is consistent (no gaps, no duplicates).
